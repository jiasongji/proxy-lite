Proxy Lite 部署运维手册

轻量 A/B 中转落地脚本：AnyTLS / NaiveProxy / Shadowsocks 三协议平级，服务器 A 与 B 都可直接提供用户入口；每个入口协议可独立选择 direct 或 egress-b 出口；Shadowsocks 同时承担 A→B 内部落地。

1. 概览 2. 入口 vs 落地 3. 安装示例 4. 路由 5. 命令 6. 升级回退 7. 审计 8. 安全

1. 概览

Proxy Lite 是完整 proxy-manager 的独立轻量项目，去掉多用户、流量限制和域名/IP 分流功能，只保留 A/B 中转落地、三协议平级用户入口与协议级出站选择。

项目值
短命令PL（大写）
完整命令proxy-lite
GitHubhttps://github.com/jiasongji/proxy-lite
Latest Releasev0.1.2
默认镜像ghcr.io/sagernet/sing-box:latest

保留 AnyTLS、NaiveProxy、Shadowsocks（用户入口 + A-B 内部落地）、Docker host network、真实 sing-box check -c、安全升级和失败回退。

裁剪 user / route / stats / traffic / quota；不创建 config/users.json；不生成 route.rule_set。

2. 用户入口 vs 内部落地

概念含义凭据导出给用户
用户入口（anytls / naive / ss）终端用户直接连接的入站协议--components + *_PORT/*_PASSWORD是
A-B 内部落地（b-ss）A 经 B 出口时的 Shadowsocks 链路：A 的 egress-b 出站、B 的 ss-landing-in 入站--b-ss-*否
同一台服务器上「用户 SS 入口」与「A-B 落地」是两套独立的 Shadowsocks 入站：端口、密码互不相同。用户入口凭据写入 config/client/；落地凭据只保存在服务端，绝不外泄。

3. 安装示例

快速安装

curl -fsSL https://github.com/jiasongji/proxy-lite/releases/latest/download/proxy-lite.sh -o /tmp/proxy-lite.sh
bash /tmp/proxy-lite.sh install

服务器 B：仅作 A-B Shadowsocks 落地

PL install --yes \
  --node-role egress_b \
  --domain b.example.com \
  --server-ip 198.51.100.20 \
  --b-ss-port 30003 \
  --b-ss-method aes-128-gcm \
  --b-ss-password '<B_SS_PASSWORD>'

服务器 B：落地 + 直接提供 Shadowsocks 入口

PL install --yes \
  --node-role egress_b \
  --domain b.example.com \
  --server-ip 198.51.100.20 \
  --b-ss-port 30003 \
  --b-ss-method aes-128-gcm \
  --b-ss-password '<B_SS_PASSWORD>' \
  --components ss \
  --ss-port 30014 \
  --ss-method aes-128-gcm \
  --ss-password '<USER_SS_PASSWORD>'

服务器 A：只部署 Shadowsocks 入口，直出

PL install --yes \
  --node-role entry_a \
  --domain a.example.com \
  --server-ip 203.0.113.10 \
  --components ss \
  --ss-egress direct \
  --ss-port 30004 \
  --ss-password '<USER_SS_PASSWORD>'

服务器 A：Shadowsocks 入口经 B 落地

PL install --yes \
  --node-role entry_a \
  --domain a.example.com \
  --server-ip 203.0.113.10 \
  --components ss \
  --ss-egress egress-b \
  --ss-port 30004 \
  --ss-password '<USER_SS_PASSWORD>' \
  --b-ss-host 198.51.100.20 \
  --b-ss-port 30003 \
  --b-ss-method aes-128-gcm \
  --b-ss-password '<B_SS_PASSWORD>'

服务器 A：三协议混合出站（AnyTLS direct / NaiveProxy 经 B / Shadowsocks direct）

PL install --yes \
  --node-role entry_a \
  --domain a.example.com \
  --server-ip 203.0.113.10 \
  --components anytls,naive,ss \
  --anytls-egress direct \
  --naive-egress egress-b \
  --ss-egress direct \
  --cert-file /path/fullchain.pem \
  --key-file /path/privkey.pem \
  --anytls-port 30001 \
  --naive-port 30002 \
  --ss-port 30004 \
  --ss-password '<USER_SS_PASSWORD>' \
  --b-ss-host 198.51.100.20 \
  --b-ss-port 30003 \
  --b-ss-method aes-128-gcm \
  --b-ss-password '<B_SS_PASSWORD>'

4. 路由

entry_a 不做域名/IP/AI split 分流，只按用户入口协议 tag 路由，route.final 恒为 direct：anytls-in / naive-in / ss-in 分别按各自的 *_EGRESS 走 direct 或 egress-b。

组合参数效果
任意协议 direct--<proto>-egress direct该入口从本机出口。
任意协议经 B--<proto>-egress egress-b + --b-ss-*该入口经 B 落地后直出。
混合各协议分别指定不同入口走不同出口。
全部 direct所有 --*-egress direct无需 B 参数，不生成 egress-b。
{
  "route": {
    "rules": [
      { "inbound": ["anytls-in"], "action": "route", "outbound": "direct" },
      { "inbound": ["naive-in"], "action": "route", "outbound": "egress-b" },
      { "inbound": ["ss-in"], "action": "route", "outbound": "direct" }
    ],
    "final": "direct"
  }
}

egress_b 的所有入口协议与落地流量都从本机 direct 出口，路由为 { "final": "direct" }。

只有至少一个入口协议选择 egress-b 时，服务器 A 才需要 --b-ss-host、--b-ss-port、--b-ss-method、--b-ss-password。

6. 升级回退

PL update
PL backup list
PL upgrade --image ghcr.io/sagernet/sing-box:latest
PL check
PL status

PL upgrade 会先备份当前配置，拉取候选镜像，重新渲染配置，并使用候选镜像执行真实 sing-box check -c。校验通过才会应用；候选镜像拉取、配置渲染或配置检查失败时，会恢复更新前快照。

PL backup list
PL rollback latest
PL rollback 20260616-120000

从 v0.1.1 升级是安全的：旧 entry_a 仍无 SS 用户入口，旧 egress_b 仍仅落地，除非显式 --components ...ss... 启用。

7. 审计

  • bash -n proxy-lite.sh 与 ShellCheck。
  • CLI help / menu smoke。
  • 生成配置矩阵：anytls/naive/ss 单协议与组合、ss direct、ss via B、三入口混合、全部 direct、全部经 B、B 仅落地、B 落地+anytls、B 落地+ss（双 SS 入站隔离）。
  • 对抗测试：经 B 缺 B 参数失败、SS 入口端口与落地端口冲突失败、SS 用户密码不外泄、旧 env 迁移不产生 SS 用户入站。
  • Docker sing-box check -c 真实检查。
  • 升级/回退 smoke：有效镜像通过，无效镜像失败并恢复快照。

8. 安全

  • B_SS_PASSWORD 是 A-B 内部落地密码，不提供给终端用户；与用户 SS 入口密码物理隔离。
  • 客户端目录按启用入口导出 AnyTLS / NaiveProxy / Shadowsocks 配置与 full test client；仅落地节点不导出。
  • 公开文档仅使用占位 IP、域名、端口和密码。
  • B 的落地端口建议在云安全组、防火墙或面板中限制为仅允许 A 的公网 IP；B 的用户入口端口需对用户开放。
